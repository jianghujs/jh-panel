# 主备状态与云监控联动施工方案

**版本**：v0.1  
**更新日期**：2026-06-30

---

## 1. 施工目标

本施工方案用于把“主备状态可视化 + 云监控采集 + 数据同步回调”落地为可开发、可联调、可验收的改造任务。

| 目标           | 验收结果                                                    |
| -------------- | ----------------------------------------------------------- |
| 主备关系可配置 | 江湖面板可绑定对端 IP、对端SSH公钥认证信息和配置角色标识    |
| 主备状态可检查 | 本机可检查自身状态，并通过 SSH 读取对端状态判断是否一主一备 |
| 云监控可展示   | 云监控主机管理和日报能展示主备角色、状态和异常原因          |
| 状态变化可回调 | 主备状态变化后，云监控按规则回调江湖数据同步                |
| 异常可兜底     | 回调失败可重试、邮件通知、人工重试；切换中超过 1 小时告警   |

## 3. 江湖面板施工项

### 3.1 新增主备管理插件

主备管理插件负责保存主备关系配置，并输出结构化状态给云监控采集。

| 项目     | 内容                                       |
| -------- | ------------------------------------------ |
| 插件名   | `ha_manager`                               |
| 中文名   | 主备管理                                   |
| 插件目录 | `/www/server/jh-panel/plugins/ha_manager/` |
| 数据目录 | `/www/server/ha_manager/`                  |
| 配置文件 | `/www/server/ha_manager/config.json`       |

### 3.2 配置字段

主备关系只绑定两台服务器。

| 字段                  | 示例                      | 说明                                                  |
| --------------------- | ------------------------- | ----------------------------------------------------- |
| `relation_id`         | `prod-db-ha-001`          | 主备关系 ID                                           |
| `local_ip`            | `x.x.x.x`                 | 本机公网 IP（插件自动识别，仅用于状态输出，无需填写） |
| `peer_ip`             | `y.y.y.y`                 | 对端公网 IP                                           |
| `ssh_user`            | `root`                    | 对端 SSH 用户                                         |
| `ssh_port`            | `22`                      | 对端 SSH 端口                                         |
| `ssh_auth_type`       | `key`                     | SSH 认证方式，优先密钥                                |
| `ssh_key_path`        | `/root/.ssh/standby_sync` | SSH 私钥路径                                          |
| `peer_ssh_public_key` | `ssh-rsa ...`             | 对端 SSH 公钥，用于授权对端访问本机                   |
| `configured_role`     | `primary` / `standby`     | 当前配置角色标识                                      |

### 3.3 绑定主备过程

绑定主备采用点对点向导的交互方式：本机展示自己的 SSH 公钥，表单填写对端公网 IP 和对端 SSH 公钥。两台服务器都完成一次配置后，就能建立双向 SSH 互信。

**向导页面布局**：

| 区域     | 内容                                           | 说明                                                         |
| -------- | ---------------------------------------------- | ------------------------------------------------------------ |
| 本机信息 | 本机 SSH 公钥、复制公钥按钮                    | 复制后粘贴到对端主备管理插件                                 |
| 对端信息 | 对端公网 IP、对端 SSH 公钥、SSH 端口、SSH 用户 | 对端公网 IP 和对端 SSH 公钥必填；端口默认 `22`，用户默认 `root` |
| 关系标识 | `relation_id`                                  | 两台服务器使用同一个主备关系 ID                              |
| 角色标识 | `configured_role`                              | 下拉选择 `primary` / `standby`                               |
| 操作按钮 | 保存配置、测试连接                             | 保存写入配置和公钥授权；测试连接验证 SSH 互通                |

**绑定流程**：

| 步骤 | 动作                                                         | 结果                                      |
| ---- | ------------------------------------------------------------ | ----------------------------------------- |
| 1    | A 服务器插件展示本机 SSH 公钥                                | 操作员复制 A 的公钥                       |
| 2    | B 服务器插件展示本机 SSH 公钥                                | 操作员复制 B 的公钥                       |
| 3    | A 服务器填写 B 的公网 IP、B 的 SSH 公钥、`relation_id` 和 `configured_role`，点击“保存配置” | A 保存对端信息，并将 B 的公钥写入本机授权 |
| 4    | B 服务器填写 A 的公网 IP、A 的 SSH 公钥、同一个 `relation_id` 和对应 `configured_role`，点击“保存配置” | B 保存对端信息，并将 A 的公钥写入本机授权 |
| 5    | 两边点击“测试连接”                                           | 插件尝试 SSH 连接对端                     |
| 6    | SSH 成功后在对端执行 `ha_manager get_status --local-only`    | 读取对端角色、切换状态和本机自检结果      |
| 7    | 本机根据本机状态 + 对端状态执行首次“一主一备”检查            | 输出 `normal/warning/error` 和异常原因    |
| 8    | 保存连接状态和最近一次检查结果                               | 供云监控采集 `ha_relation` 字段           |

**绑定列表**：插件页面展示当前主备关系列表，每行显示对端公网 IP、`relation_id`、`configured_role` 和连接状态。连接状态列有刷新按钮，点击后重新 SSH 连接对端并更新状态。

连接状态取值：

| 状态                   | 说明                         |
| ---------------------- | ---------------------------- |
| `pending`              | 只完成本边配置，尚未测试连接 |
| `connected`            | SSH 连接成功，可读取对端状态 |
| `ssh_timeout`          | SSH 连接超时                 |
| `ssh_auth_failed`      | SSH 认证失败                 |
| `peer_plugin_missing`  | 对端未安装主备管理插件       |
| `relation_id_mismatch` | 对端 `relation_id` 不一致    |

如果只完成单边配置，连接状态为 `pending`，页面提示“等待对端保存本机公钥后测试连接”。

### 3.4 插件方法

插件至少提供两个方法：读取状态和切换角色。

| 方法          | 命令                                                         | 用途                           |
| ------------- | ------------------------------------------------------------ | ------------------------------ |
| `get_status`  | `python3 /www/server/jh-panel/plugins/ha_manager/index.py get_status` | 输出本机主备状态和对端检查结果 |
| `switch_role` | `python3 /www/server/jh-panel/plugins/ha_manager/index.py switch_role <role>` | 由切换脚本写入角色状态         |

`switch_role` 支持的状态：

| 参数        | 写入效果                                                |
| ----------- | ------------------------------------------------------- |
| `switching` | 写入 `switch_state=switching`，记录 `switch_started_at` |
| `primary`   | 写入 `configured_role=primary`，`switch_state=normal`   |
| `standby`   | 写入 `configured_role=standby`，`switch_state=normal`   |
| `failed`    | 写入 `switch_state=failed`，记录失败时间                |

### 3.5 主备自检

主备管理插件提供自检能力，检查本机状态并通过 SSH 读取对端状态，综合判断“一主一备”是否正常。

**自检页面**：插件页面增加“自检结果”区域，展示当前检查项状态和异常原因，支持手动触发自检。

**检查项配置**：插件页面增加“检查项设置”区域，可启用或禁用各检查项。

| 检查项           | 默认启用 | 检查逻辑                                                   | 异常输出                                               |
| ---------------- | -------- | ---------------------------------------------------------- | ------------------------------------------------------ |
| 本机角色一致性   | 是       | 对比 `configured_role` 与实际运行状态                      | `role_mismatch`                                        |
| MySQL 主从状态   | 是       | 检查主从同步是否正常、从库延迟                             | `mysql_replication_error` / `mysql_replication_lag`    |
| MySQL 读写与角色 | 是       | 主机应可写、备机应只读                                     | `mysql_write_on_standby` / `mysql_readonly_on_primary` |
| rsync 同步方向   | 是       | 主机应推送到备机                                           | `rsync_direction_error`                                |
| rsync 延迟       | 是       | 检查同步延迟和待同步文件量                                 | `rsync_lag`                                            |
| 对端 SSH 状态    | 是       | SSH 到 `peer_ip` 执行 `ha_manager get_status --local-only` | `peer_unreachable` / `peer_status_error`               |
| 一主一备         | 是       | 本机和对端必须只有一个 `primary`、一个 `standby`           | `ha_role_conflict`                                     |
| Keepalived VIP   | 否       | 启用后检查 VIP 是否在主机侧                                | `vip_on_wrong_host`                                    |
| 切换中超时       | 是       | `switching` 超过 1 小时                                    | `switching_timeout`                                    |

**对端检查规则**：

| 场景                                | 处理                                                  |
| ----------------------------------- | ----------------------------------------------------- |
| 本机云监控和对端云监控不在同一实例  | 不依赖云监控聚合，插件直接 SSH 到对端检查             |
| 对端 SSH 不通                       | 本次自检输出 `warning`，保留失败原因和时间            |
| 对端插件不存在                      | 输出 `peer_plugin_missing`，提示先安装主备管理插件    |
| 对端返回 `switching`                | 一主一备检查暂不判死，展示“切换中”；超过 1 小时后告警 |
| 两端都为 `primary` 或都为 `standby` | 输出 `ha_role_conflict`，云监控按异常展示             |

`get_status` 默认输出完整检查结果；插件内部 SSH 调用对端时使用 `--local-only`，避免两端互相递归检查。

## 4. 切换脚本施工项

切换脚本统一通过主备管理插件写状态，避免每个脚本直接改配置文件。

### 4.1 新增一键切换脚本

新增一键主备切换脚本，用于统一执行主降备、备升主、同步方向调整和状态写入。

| 步骤 | 动作                                                         |
| ---- | ------------------------------------------------------------ |
| 1    | 调用 `switch_role switching`                                 |
| 2    | 执行 MySQL、rsync、任务、入口服务等切换动作                  |
| 3    | 成功后按目标角色调用 `switch_role primary` 或 `switch_role standby` |
| 4    | 失败时调用 `switch_role failed`，并保留脚本日志              |

### 4.2 修改旧脚本

旧脚本继续保留，但必须补充状态写入。

| 脚本               | 开始时                  | 成功时                | 失败时               |
| ------------------ | ----------------------- | --------------------- | -------------------- |
| `notify_master.py` | `switch_role switching` | `switch_role primary` | `switch_role failed` |
| `notify_backup.py` | `switch_role switching` | `switch_role standby` | `switch_role failed` |

`report_collector.py` 采集 Debian 状态时增加 `ha_relation` 字段。

| 字段              | 说明                      |
| ----------------- | ------------------------- |
| `relation_id`     | 主备关系 ID               |

| `configured_role` | 配置角色标识              |
| `actual_role`     | 实际判断角色              |
| `switch_state`    | `normal/switching/failed` |
| `peer`            | 对端 SSH 检查结果         |
| `checks`          | 检查项列表                |
| `summary_status`  | `normal/warning/error`    |
| `summary_msg`     | 状态摘要                  |

### 5.2 主机配置增加 server_key

云监控主机配置增加 `server_key`，用于江湖数据同步绑定数据库和同步任务。

| 字段         | 示例                | 说明                         |
| ------------ | ------------------- | ---------------------------- |
| `server_key` | `prod-db-primary-a` | 人工配置的稳定服务器唯一标识 |

### 5.3 页面展示

| 页面     | 展示内容                                   |
| -------- | ------------------------------------------ |
| 主机管理 | 主备角色、关系 ID、对端 IP、状态、异常原因 |
| 单机日报 | 主备状态、检查结果、切换中/失败提示        |
| 总览日报 | 主备异常主机汇总                           |

### 5.4 状态回调

云监控设置页增加主机状态回调配置。

| 配置项   | 说明                                           |
| -------- | ---------------------------------------------- |
| 回调地址 | 主备状态变化时通知的 URL                       |
| 签名密钥 | 用于回调验签                                   |
| 回调记录 | 设置旁边弹框查看请求、响应、失败原因和重试次数 |

重试规则：

| 次数    | 时间                 | 停止条件     |
| ------- | -------------------- | ------------ |
| 第 1 次 | 状态变化后立即回调   | 接口返回完成 |
| 第 2 次 | 第一次失败后 5 分钟  | 接口返回完成 |
| 第 3 次 | 第二次失败后 10 分钟 | 接口返回完成 |

三次失败后发送邮件通知，失败记录支持在回调记录弹框中手动重试。

## 6. 江湖数据同步施工项

江湖数据同步通过 `server_key` 绑定数据库和同步任务，不直接依赖云监控内部 `host_id`。

| 字段                 | 说明               |
| -------------------- | ------------------ |
| `database_key`       | 数据库标识         |
| `primary_server_key` | 主服务器标识       |
| `standby_server_key` | 备服务器标识       |
| `active_server_key`  | 当前生效服务器标识 |
| `sync_task_key`      | 同步任务标识       |
| `switch_policy`      | `auto/manual`      |

回调处理流程：

1. 校验签名和 `event_id` 幂等。
2. 根据 `current_host.server_key` 和 `peer_host.server_key` 查数据库绑定关系。
3. 如果当前主机变为 `primary`，将对应数据库切到当前服务器并启用同步任务。
4. 如果当前主机变为 `standby`，将对应数据库标记为备库或目标库。
5. 返回 `status=success` 或 `handled=true` 后，云监控停止重试。
