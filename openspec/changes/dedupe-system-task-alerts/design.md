## Context

`task.py` 中的 `systemTask()` 每 5 秒采样一次系统状态，并调用 `mw.generateMonitorReportAndNotify(...)` 生成异常通知。当前实现把所有监控异常汇总后，通过 `mw.notifyMessage(..., stype='面板监控', trigger_time=600)` 发送。

这意味着当前抑制重复通知的粒度是“通知类型 + 时间窗口”，而不是“具体问题实例”。只要异常持续存在，就会在时间窗口结束后再次发送；反过来，如果不同的新问题在同一个时间窗口内先后出现，也可能被同一个 `stype` 节流掉。

用户希望的行为是：同一问题在持续异常期间只发送一次，只有在问题恢复后再次出现时才重新发送。这要求监控路径具备跨轮询的状态记忆，并能区分“新出现的问题”“仍然存在的问题”“已经恢复的问题”。

## Goals / Non-Goals

**Goals:**
- 为 `systemTask` 监控异常建立稳定的问题标识，并按问题维度去重
- 仅在问题首次出现时发送通知，不在持续异常期间重复发送
- 在问题恢复后清理状态，使再次异常时能够重新通知
- 保持现有通知渠道、配置入口和大部分消息格式不变
- 避免不同问题之间因为共享 `stype='面板监控'` 而互相节流

**Non-Goals:**
- 不修改 Telegram / 邮件通知配置结构
- 不为本次变更增加“恢复通知”消息
- 不统一重构所有 `mw.notifyMessage` 调用方，只处理 `systemTask` 监控告警路径
- 不改变监控阈值判断逻辑本身，只调整告警发送时机

## Decisions

### 1. 在 `generateMonitorReportAndNotify` 路径引入“活动告警状态”文件
- 决策：新增一份本地 JSON 状态文件，记录当前仍处于异常中的问题键集合以及必要的元数据。
- 推荐位置：`data/system_task_alert_state.json`。
- 原因：
  - `data/` 比 `tmp/` 更适合保存跨 reload / 重启后的监控状态，避免服务重启后对同一持续故障再次误报。
  - 状态文件结构简单，不需要引入数据库或新的表结构。
- 备选方案：
  - 仅使用内存状态：实现简单，但 reload 后会丢失状态并重复报警。
  - 继续依赖 `notify_lock.json` + 更细粒度 `stype`：难以表达“恢复后再发”，也会把通知系统的通用节流机制和监控问题生命周期混在一起。

### 2. 为每类监控异常生成稳定的问题键
- 决策：在构建异常列表时，同时产出 `problem_key`，并以该键作为去重和恢复判断的唯一标识。
- 示例：
  - CPU: `cpu-high`
  - Memory: `memory-high`
  - Disk root usage: `disk-usage:/`
  - SSL cert: `ssl-cert:<site-name>`
  - MySQL slave: `mysql-slave:<slave-ip-or-index>`
  - Rsync realtime: `rsync-realtime`
  - Keepalived: `keepalived-service`
- 原因：
  - 仅按消息文本对比会受动态数值、时间字段影响，导致同一问题被误判为新问题。
  - 问题键可以让“问题定义”和“通知文本”解耦。
- 备选方案：
  - 直接对消息内容做哈希：实现快，但文本格式一变就会打破去重稳定性。

### 3. 每轮计算 `current_active_keys`，用集合差异驱动发送与恢复
- 决策：每次轮询时构建当前活跃问题集合，与上次保存的活动集合比较：
  - `new_keys = current - previous`：这些问题需要发送通知
  - `recovered_keys = previous - current`：这些问题需要从状态中移除
  - `continued_keys = current ∩ previous`：这些问题保持静默
- 同时，只有 `new_keys` 对应的消息会被加入本轮发送内容，避免在发送新问题时顺带重复发送旧问题。
- 原因：这正好对应用户要求的“首次出现发送、恢复后再出现才重发”。
- 备选方案：
  - 每个问题单独设置时间窗口：仍然无法准确表达恢复边界。

### 4. `面板监控` 路径不再依赖 `trigger_time=600` 作为主去重机制
- 决策：监控告警发送时保留 `stype='面板监控'`，但应将 `trigger_time` 设为不会阻挡新问题的值，例如 `0`，把去重职责完全交给问题状态比较逻辑。
- 原因：如果继续使用 600 秒共享节流，不同新问题在 10 分钟内先后出现时，后者可能被错误吞掉。
- 备选方案：
  - 为不同问题动态拼接 `stype`：会把监控领域逻辑侵入通用通知锁机制，长期维护成本更高。

### 5. 状态文件读写失败时按“空状态”降级，但必须覆盖写回
- 决策：若状态文件不存在、为空或 JSON 损坏，则以空状态启动本轮比较，并在本轮结束后重写合法内容。
- 原因：保证功能可恢复，最坏情况只是异常持续时多发一次，而不是彻底失去告警能力。

## Risks / Trade-offs

- [状态文件损坏或被人工删除] -> 视为无活动状态并重建文件，可能导致当前持续异常重新发送一次
- [问题键设计不稳定] -> 为每类异常选择与实体绑定的固定键，不使用含时间戳或瞬时数值的动态文本
- [仅发送新问题会让通知内容不再包含全部当前异常] -> 这是有意的取舍，用于避免旧问题被重复报警；完整状态仍可通过面板监控页查看
- [持久化到 `data/` 后，长时间未恢复的问题会跨进程保留] -> 这符合“未恢复不重发”的目标，但需要保证恢复逻辑可靠清理状态

## Migration Plan

1. 新增状态读写辅助逻辑，并在无旧文件时自动初始化为空状态
2. 调整 `generateMonitorReportAndNotify` 的异常收集结构，从纯文本列表升级为“问题键 + 文本”的列表
3. 仅对 `new_keys` 生成并发送通知，对恢复问题做状态清理
4. 上线后观察 `logs/notify.log` 和状态文件变化，确认持续异常不再重复发送
5. 如需回滚，可移除状态比较逻辑并恢复原来的 `trigger_time=600` 发送方式；新状态文件可忽略或删除

## Open Questions

- 状态文件是否还需要记录 `first_seen` / `last_seen` 以便后续调试；本次实现至少需要保存活动键集合，附加时间戳可作为可选增强
