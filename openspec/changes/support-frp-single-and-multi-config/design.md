## Context

当前 frp 插件的后端与前端都把配置入口硬编码为 `frps.ini` 和 `frpc.ini` 两个文件：

- `index.py` 只暴露固定的服务端/客户端配置路径与模板列表。
- `index.html` 通过通用 `pluginConfigTpl(...)` 入口直接编辑单文件内容。
- `init.d` 和 `systemd` 模板也固定使用 `frps -c frps.ini` 与 `frpc -c frpc.ini` 启动。

这意味着插件无法同时覆盖以下两类运维习惯：

- 继续使用当前“单文件集中维护”的老用户；
- 希望按业务拆分为多个配置文件分别维护的新用户。

本次变更既涉及 UI 入口，也涉及配置存储、服务启动、自启动模板和升级兼容，属于跨模块调整，适合先明确设计。

## Goals / Non-Goals

**Goals:**

- 让 frp 插件同时支持单配置文件模式和多配置文件模式。
- 保持现有 `frps.ini` / `frpc.ini` 用户升级后默认可继续使用，无需强制迁移。
- 允许服务端和客户端分别选择模式，不要求两端必须一致。
- 为多配置模式提供可枚举、可编辑、可新增、可删除的配置文件管理能力。
- 让启动、自启动和状态检测逻辑与所选模式保持一致。

**Non-Goals:**

- 不改造 frp 二进制安装流程和版本来源。
- 不引入复杂的配置向导、代理规则可视化编排或跨文件依赖分析。
- 不自动把单文件内容拆分迁移成多个配置文件；仅提供兼容和手工切换能力。
- 不解决 frp 配置语义本身的正确性问题（例如端口冲突、业务规则冲突），仅做基础文件级校验。

## Decisions

### 1. 用显式模式元数据而不是目录推断

为 frp 插件增加模式元数据文件，例如 `server/frp/config_mode.json`，至少包含：

- `server_mode`: `single` | `multi`
- `client_mode`: `single` | `multi`

理由：

- 升级后若同时存在旧单文件和新目录，系统无需猜测该使用哪一套。
- 服务端和客户端可以独立切换，不互相影响。
- 前端可以稳定展示当前模式，避免“目录空/非空”导致 UI 状态跳变。

备选方案：

- 仅依据 `frps.d/`、`frpc.d/` 目录是否存在或是否有文件自动判定。
  这个方案实现更省，但在回滚、混用或升级中存在歧义，因此不采用。

### 2. 保留旧单文件路径，新增多文件目录

单文件模式继续使用：

- `server/frp/frps.ini`
- `server/frp/frpc.ini`

多文件模式新增目录：

- `server/frp/frps.d/*.ini`
- `server/frp/frpc.d/*.ini`

理由：

- 对现有用户零迁移成本。
- 文件布局直观，易于区分服务端/客户端配置集合。
- 与现有插件模板目录结构和面板语义容易对应。

备选方案：

- 统一改成新的根文件名或统一目录结构。
  该方案会增加升级成本，并破坏现有外部脚本依赖，因此不采用。

### 3. 多文件模式按“一个配置文件一个进程”运行

在多文件模式下，`frps` 与 `frpc` 不再假设单一入口文件，而是遍历各自目录中的 `*.ini` 文件，为每个配置文件启动一个独立进程；单文件模式仍保持现状。

理由：

- 不依赖特定 frp 版本是否支持 include/merge 语法。
- 配置文件天然隔离，便于按实例管理和排查。
- 与当前 `.ini` 模板体系兼容，不需要整体切换到新格式。

备选方案：

- 生成一个聚合主配置，再通过 include 引用多个子文件。
  该方案对 frp 配置语法和版本能力有额外假设，兼容性更弱，因此不采用。

### 4. 启停逻辑改成统一 wrapper，systemd/init.d 都复用

新增统一的运行包装脚本，负责：

- 读取模式元数据；
- 决定是启动固定单文件还是遍历多文件目录；
- 记录 pid 或按配置文件名管理进程；
- 在 stop/restart/status 时按同一规则处理。

`systemd` 与 `init.d` 都调用这套 wrapper，而不是把单文件路径直接写死在模板中。

理由：

- 避免在多个服务模板里重复维护模式判断逻辑。
- 后续若增加日志、pid、锁文件规则，只需要修改一处。
- 便于同时兼容 Linux systemd 和 Apple/旧式 init.d 分支。

备选方案：

- 继续在 `frpc.service.tpl` / `frps.service.tpl` 里内联模式判断。
  该方案模板会迅速变复杂，且两端逻辑容易漂移，因此不采用。

### 5. 前端从“单一编辑器入口”改为“模式感知的配置面板”

前端不再仅依赖通用 `pluginConfigTpl(...)` 打开固定文件，而是新增 frp 自己的配置管理 UI，至少支持：

- 查看服务端/客户端当前模式；
- 切换 `single` / `multi` 模式；
- 单文件模式下继续编辑固定配置；
- 多文件模式下列出配置文件、选择模板新建、编辑、删除。

理由：

- 通用单文件编辑入口无法覆盖多文件列表和模式切换。
- frp 的服务端/客户端模板本就分开，适合做成专用交互。

备选方案：

- 保留旧入口，只额外增加一个“多配置目录说明”。
  该方案无法真正管理多个配置文件，因此不采用。

## Risks / Trade-offs

- [模式切换后两套配置并存] → 默认仅按元数据指定模式启动，未激活模式的文件保留但不参与运行；UI 明确提示当前生效模式。
- [多进程启动后 stop/status 粒度变复杂] → wrapper 统一维护 pid 文件或按配置文件名匹配命令行，避免直接 `kill` 全部 `frp` 相关进程。
- [旧服务模板升级后仍残留写死单文件路径] → 安装/升级流程中重建或覆盖 frp 相关 service/wrapper 文件，并执行 `daemon-reload`。
- [用户切到 multi 模式但目录为空] → 前端在首次切换时自动创建一个默认配置文件，或提示先新建配置后再启动。
- [多个配置文件之间端口或代理名冲突] → 保存时做基础重名/空文件校验，运行失败信息回显到面板；不承诺做完整语义冲突分析。

## Migration Plan

- 升级时如果 `config_mode.json` 不存在，则自动初始化为 `server_mode=single`、`client_mode=single`。
- 保留现有 `frps.ini` / `frpc.ini` 与模板读取能力，确保老用户升级后无需修改即可继续运行。
- 新增 `frps.d/` / `frpc.d/` 目录，但不自动迁移旧配置内容。
- 更新 service/wrapper 后，通过手动验证覆盖以下路径：
  - 旧用户不切模式直接启动；
  - 切到 multi 后新增配置并启动；
  - 从 multi 切回 single 后仍能恢复单文件启动。
- 若上线后需要回滚，删除或忽略 `config_mode.json` 与多文件目录即可回到旧单文件行为；旧单文件内容保持不变。

## Open Questions

- 多文件模式下是否需要允许“批量启用/停用某个配置文件”，还是默认目录内全部 `*.ini` 一起启动即可？
- 是否需要为多文件配置增加排序字段或前缀约定，以便控制启动顺序？
- 多文件模式下的日志是否保持服务端/客户端共用，还是按配置文件拆分单独日志文件？
